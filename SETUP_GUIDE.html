2Install the project

From the project root (the folder containing package.json):

npm install

This installs Next.js 14, Tailwind, the Supabase SDK and everything else. It takes a minute or two the first time.

Windows note All commands work in PowerShell, Git Bash or the VS Code terminal. Use forward slashes in paths.

3Create the Supabase project

  1. Go to the Supabase dashboard and click New project.
  2. Pick a name, a strong database password, and a region close to your users (London / Frankfurt for UK & DE — also helps GDPR).
  3. Wait ~2 minutes for it to provision.
  4. Open Project Settings → API and copy these three values — you'll paste them into .env.local in step 6:
    • Project URL → NEXT_PUBLIC_SUPABASE_URL
    • anon / public key → NEXT_PUBLIC_SUPABASE_ANON_KEY
    • service_role key → SUPABASE_SERVICE_ROLE_KEY keep secret
Never expose the service_role key to the browser. It bypasses Row Level Security and is only used server-side (cron jobs, webhooks, the seed script).

4Apply the database schema

The schema lives in supabase/migrations/ as numbered files (0001–0015). They must run in order — the helper functions in 0001 are used by every table's security policy. This also creates the Storage buckets automatically. The final file, 0015_realtime.sql, adds the dispatch-board tables to the Realtime publication so the live schedule updates without polling.

Option A — Supabase CLI (recommended)

# Install the CLI once (https://supabase.com/docs/guides/cli)
npm install -g supabase

# Link to your project (find the ref in the dashboard URL)
supabase link --project-ref <your-project-ref>

# Push every migration in order
supabase db push

Option B — SQL Editor (no CLI)

  1. In the dashboard open SQL Editor → New query.
  2. Open each file in supabase/migrations/ in ascending order (0001_helpers.sql first), paste its contents, and click Run.
  3. Repeat through 0015_realtime.sql.
Verify it worked In Table Editor you should now see companies, team_members, customers, jobs, estimates, invoices and more. In Storage you should see the buckets logos, pdfs, job-photos, voice-recordings, signatures.

5Enable Google login optional

Email/password works out of the box. To also offer "Continue with Google":

  1. In Supabase: Authentication → Providers → Google, toggle it on.
  2. Create an OAuth client in the Google Cloud Console and paste its Client ID and Client secret into Supabase.
  3. Add the redirect URL Supabase shows you to the Google client's "Authorized redirect URIs".

The app's own callback route is /auth/callback — you'll register that in step 10.

Good to know lib/env.ts reads server secrets lazily and throws a clear error only when a still-unconfigured feature is actually used — so a missing Stripe key won't break the rest of the app. Integration-backed features (Twilio voice/SMS/WhatsApp, OpenAI Whisper voice reports, Google Maps routing, Resend email, Anthropic) all degrade gracefully: without the key the feature shows a friendly "not available" message or falls back (e.g. typing a report instead of recording, a deterministic fallback narrative instead of an AI one), rather than crashing.
The Twilio voice number is set per company, not in env After adding the Twilio keys, each company sets its own inbound number in Settings → Lead capture (stored as companies.twilio_voice_number) — that's how an inbound call is matched to the right company when you have more than one.

6bTechnician PWA automatic

The technician app at /tech/* is an installable Progressive Web App (built with next-pwa). No extra configuration is needed:

  • The service worker (public/sw.js) is generated automatically by npm run build and is disabled in npm run dev so it doesn't interfere with hot reload — test offline behaviour against a production build (npm run build && npm run start).
  • Technicians install it via their browser's "Add to Home Screen"; the manifest lives at public/manifest.json.
  • Today's jobs stay viewable offline, and photo/voice uploads queue locally (IndexedDB) and retry automatically when the signal returns.

9Deploy to Vercel

  1. Push the repo to GitHub (or GitLab/Bitbucket).
  2. In Vercel, click Add New → Project and import the repository. Vercel auto-detects Next.js — keep the defaults (Build command next build, output handled automatically).
  3. Under Settings → Environment Variables, add the same keys from your .env.local. At minimum the three Supabase keys, NEXT_PUBLIC_APP_NAME and CRON_SECRET.
  4. Set NEXT_PUBLIC_APP_URL to your real deployment URL, e.g. https://app.fieldos.ai (or the *.vercel.app URL).
  5. Click Deploy.
Apply migrations to your production Supabase project too. Use a dedicated Supabase project for production (separate from local dev) and run the same migrations against it.

10Post-deploy checklist

In your production Supabase project, go to Authentication → URL Configuration and set:

  • Site URL: your deployment URL (e.g. https://app.fieldos.ai).
  • Redirect URLs: add https://your-domain/auth/callback (and your local http://localhost:3000/auth/callback for dev).

Then verify:

  • Register a new account → you're taken into the onboarding wizard.
  • Complete onboarding → you land on the dashboard.
  • Create a customer and a property → they appear and persist.
  • Log out and back in → session works; Google login works (if enabled).

12Troubleshooting

"Missing required environment variable …"

A feature tried to use a key that isn't set. Add it to .env.local (local) or Vercel env vars (prod) and restart/redeploy.

Login works but no data loads / "Could not load …"

Almost always the migrations weren't applied (or were applied out of order) against the Supabase project the app points to. Re-check step 4 and confirm the tables exist.

Redirected to login in a loop after signing in

Check the Supabase Site URL and Redirect URLs (step 10) match your actual domain, and that NEXT_PUBLIC_APP_URL is correct.

The live schedule doesn't update in real time

Confirm 0015_realtime.sql was applied — it adds appointments, jobs and team_members to the Supabase Realtime publication. Without it the dispatch board still works but won't auto-refresh.

npm run build runs out of memory locally

On lower-memory machines, give Node a bigger heap: NODE_OPTIONS="--max-old-space-size=6144" npm run build. Vercel's build environment has enough memory by default.

Offline uploads in the technician app aren't testing as expected in dev

The service worker is disabled in npm run dev. Test PWA/offline behaviour against a production build: npm run build && npm run start.

npm run seed exits with "Set NEXT_PUBLIC_SUPABASE_URL…"

Your .env.local still has the placeholder URL. Paste the real Supabase values first.

Build fails on an ESLint "unused variable" error

The production build is lint-gated. Remove the unused import/variable it names, then rebuild.

Language is wrong, or won't switch back

The UI locale follows a NEXT_LOCALE cookie, not the URL. Switch it any time from the language item in the user menu (top right) — it toggles immediately. Settings → Company sets the same cookie and is also the default new team members get at login.

Translations are only partial in German

By design: the main office list pages (leads, calls, schedule, materials, finance, coach, reviews, team) are translated. Detail pages, dialogs, status-badge labels, and the entire technician PWA (/tech/*) are still English-only — a full pass is tracked as outstanding work, not a bug.
