## Project overview

Build a production-ready, mobile-first SaaS platform called **FieldOS AI**.

**One-line pitch:** "Run your entire service company from one AI-powered platform."

FieldOS AI is an AI operating system for field service companies вЂ” not just another job-management tool like Jobber, Housecall Pro, or ServiceTitan. Those tools digitise the paperwork. FieldOS AI runs the business: it answers the phone, books the job, schedules the technician, writes the estimate, chases the invoice, and tells the owner why profit was down last month вЂ” all through one connected platform with an AI layer running through every module.

**Target trades:** plumbing, HVAC, electrical, roofing, landscaping, cleaning, pest control, appliance repair, pool services, general contracting.

**Target company size:** 1вЂ“100 employees вЂ” this explicitly includes solo operators but is built to scale up to multi-technician teams with dispatch and routing needs. The UI must work equally well for a one-person business and a twenty-technician one (modules like scheduling and team management simply have nothing to show / are hidden for solo accounts).

**Target markets:** UK and Germany at launch (English and German UI, GBP and EUR billing, UK and DE VAT rules), architected so additional markets can be added without a rebuild.

**v1 scope:** every module described in this document ships in v1, with one explicit exception вЂ” the customer-facing marketplace (matching jobs to contractors, "Uber for trades") is **out of scope for v1**. It requires two-sided liquidity (contractors AND customers already on the platform in volume) that does not exist until the core SaaS has real adoption. Building it before that point would mean building a matchmaking product with nothing to match. It is documented at the end of this file as a deliberate Phase 2, not forgotten вЂ” just sequenced correctly.

- **Real-time updates**: Supabase Realtime (live dispatch board, live lead inbox, live job status updates on the technician app)

- **Maps UI rendering**: react-map-gl (Mapbox GL) for the dispatch board map вЂ” visually nicer and cheaper for high-frequency client-side rendering than embedding Google Maps JS directly; routing/traffic computation still uses Google's APIs server-side, the map is just the rendering layer

## Core design principle: two distinct UI surfaces, one design language

This product has three very different users looking at three different surfaces, and conflating them is the most common way these platforms become cluttered and hated. Keep them structurally separate:

1. **The Office App** (`/app/...`) вЂ” what the owner, dispatcher, and office staff use. Desktop-first but fully responsive. Data-dense where it needs to be (dispatch board, finance), but every individual screen still follows the "one primary action per screen" rule below.
2. **The Technician App** (`/app/tech/...`) вЂ” what field technicians use, almost always on a phone, often one-handed, often with dirty/gloved fingers. Mobile-first to the point of mobile-only in practice: large tap targets (minimum 48px), minimal text entry, voice and photo over typing wherever possible, big obvious primary buttons, high contrast for outdoor sunlight readability.
3. **The Customer Surfaces** (`/portal/...`, `/quote/[token]`, `/invoice/[token]`) вЂ” what the end customer sees: quote acceptance, the client portal, payment pages. No login complexity, no clutter, no upsells in their face вЂ” just "here is what you asked for, here is the price, here is the button."

Shared design tokens (colour, type scale, spacing, shadcn/ui component set) apply across all three, but information density and interaction patterns differ deliberately by surface. Do not reuse the dense office dashboard layout on the technician app, and do not reuse the technician app's big-button minimalism on the office finance pages.

**Ease-of-use rules that apply everywhere:**
- Every primary screen has one obvious next action, visually dominant (size, colour, position) over everything else on the screen.
- Never make a user choose between more than 4 options in a single decision point without grouping or progressive disclosure.
- Every async action shows a loading state with human language ("Booking the job...", "Working out the best route..."), never a bare spinner with no text.
- Every empty state explains what will appear there and, where relevant, gives a single button to create the first item.
- Destructive actions always require a confirmation step that names the specific thing being destroyed ("Delete invoice INV-2026-0042?" not "Are you sure?").

## Database schema вЂ” implement exactly this in Supabase

Run these migrations in order. Enable Row Level Security on every table. This schema is multi-user from day one (unlike a single-owner schema) вЂ” a company has a `team_members` table linking multiple `auth.users` to one `companies` row with a role, and almost every other table hangs off `company_id`.

### Helper functions (create first вЂ” everything else's RLS depends on these)

```sql
-- Returns every company the current user belongs to (as owner or accepted team member)
create or replace function get_user_company_ids()
returns setof uuid as $$
  select company_id from team_members
  where user_id = auth.uid() and invite_accepted_at is not null
$$ language sql security definer stable;

-- Returns the current user's role within a specific company
create or replace function get_user_role(p_company_id uuid)
returns text as $$
  select role from team_members
  where user_id = auth.uid() and company_id = p_company_id and invite_accepted_at is not null
  limit 1
$$ language sql security definer stable;

create or replace function update_updated_at()
returns trigger as $$
begin
  new.updated_at = now();
  return new;
end;
$$ language plpgsql;
```

## Feature specifications вЂ” build each module completely

### Module 1: CRM

- Customer list (`/customers`) is searchable by name, phone, email, or property address.
- Customer detail page shows a single chronological timeline merging estimates, jobs, invoices, and calls вЂ” query each table filtered by `customer_id`, merge and sort by `created_at` in the API layer, do not try to do this with a single SQL union view in v1 (keep it simple, optimise later if needed).
- **AI customer summary** вЂ” regenerate `customers.ai_summary` via a queued job whenever a new job or invoice is added to that customer. Prompt: *"Summarise this customer's history for a tradesperson in 2-3 sentences: how many jobs, total spend, how reliably they pay, and anything notable (recurring issue, property type, communication preference). Be concise and practical, not generic."* Feed it the timeline data as structured JSON, not free text.
- **Lifetime value**: recalculated as `sum(invoices.total_inc_vat where status = 'paid' and customer_id = X)`, stored denormalised on `customers.lifetime_value` for fast list-page sorting, updated via a database trigger on invoice status change rather than recalculated on every page load.
- **Upsell suggestions**: shown as a small panel on the customer detail page, generated by a simple rule engine (not pure AI) cross-referenced with `renewal_plans` and trade-category logic вЂ” e.g. an HVAC customer with no active maintenance plan and a job older than 10 months gets "Suggest an annual service plan." Keep this rule-based and fast; only use Claude to phrase the suggestion text naturally, not to decide whether to show it.

### Module 2: AI Lead Capture

This module is the multi-channel inbox plus the AI voice receptionist вЂ” build on the same design proven in TradeQuote AI's extended features, adapted for a `leads`/`calls` schema that supports `assigned_to` (team routing) since this product has multi-technician accounts.

**Channels feeding the `leads` table:** phone calls (via the AI receptionist below), WhatsApp/SMS (inbound Twilio webhook, matched to existing customers by phone number or creates a new lead), the embeddable website widget (same pattern as TradeQuote's Feature 13 вЂ” public `/api/widget/submit` endpoint, rate-limited, validated against a `widget_public_key`), and manual entry.

**AI Voice Receptionist** вЂ” identical mechanism to TradeQuote AI's Feature 11 (Twilio Voice webhook в†’ `<Gather input="speech">` conversational loop в†’ slot-filling system prompt collecting issue, address, and urgency в†’ call summarisation в†’ lead creation), with two additions specific to this product:
- If the company has multiple technicians with defined `skills`, the slot-filling prompt also asks a clarifying question to help route the job (e.g. for HVAC: "is this a heating or cooling issue?") so the resulting lead can later suggest the right technician at scheduling time.
- For Pro/Enterprise tier companies, allow a custom greeting and business name pronunciation override in settings (text field, used directly in the `<Say>` greeting).

**Lead scoring** вЂ” automatic, runs on lead creation via the queue (`jobs_queue` task `score_lead`):
```
System prompt: You are scoring inbound leads for a {trade} business. Given the lead's
message/transcript, classify as "hot", "warm", or "cold":
- hot: urgent/emergency language, specific address given, ready-to-proceed signals
  ("need this today", "how soon can you come")
- warm: clear job description but no urgency, or vague on timing
- cold: vague inquiry, explicit price-shopping language ("just getting quotes around"),
  or insufficient detail to act on
Respond with ONLY JSON: {"score": "hot|warm|cold", "reason": "one short sentence"}
```
- `hot` and `warm` leads trigger an immediate notification (push + WhatsApp digest); `cold` leads are batched into the existing daily digest cron only вЂ” do not interrupt the owner for every cold lead.

**Lead routing**: if the company has technicians with `skills` matching the lead's `trade_category`/inferred issue type, suggest (do not auto-assign) the best-fit technician on the lead detail page as a one-click "Assign to {name}" button.

### Module 3: Smart Scheduling

This is the most technically demanding module вЂ” build it on top of real routing infrastructure, not a custom solver.

**Dispatch board (`/schedule`)**: a calendar/timeline view (one row per technician, day or week granularity) plus a live map. Use Supabase Realtime subscriptions on `appointments` and `team_members.last_known_lat/lng` so the board updates live without polling. Technician location updates: the technician app pings `PATCH /api/tech/location` every 60 seconds while a shift is active (only while the technician app is open and the technician has granted location permission вЂ” never track in the background without explicit, visible consent, and show a persistent "Location sharing: ON" indicator in the tech app while it's active).

**AI Scheduler (`POST /api/schedule/auto-schedule`)**:
- Input: an unscheduled `job_id`.
- Logic (deterministic, not a black box): (1) filter technicians by matching `skills` to the job's `trade_category`, (2) filter by availability in their existing `appointments` for the target day, (3) for each remaining candidate, call the Google Distance Matrix API to estimate travel time from their previous appointment's location (or home base if first job of the day) to this job's property, (4) score candidates by a simple weighted formula: `score = (skill_match_weight) - (travel_time_minutes * travel_penalty) - (overtime_risk_penalty if this pushes them past business_hours)`, (5) propose the top-scoring technician and time slot, shown to the dispatcher as a suggestion card with a one-tap "Confirm" вЂ” never auto-book without a human confirming, in v1.
- This is intentionally a transparent scoring formula, not an opaque ML model вЂ” dispatchers need to trust and understand why a suggestion was made.

**Route optimization (`POST /api/schedule/optimize-route`)**:
- For a technician's full day of appointments, call the Google Routes API's optimization endpoint (`computeRoutes` with `optimizeWaypointOrder: true`) to reorder stops for minimum total drive time, respecting each appointment's fixed time window if the customer was given one.
- Update `appointments.route_order` and `travel_time_minutes_from_previous` with the result. Show the optimized order on the dispatch board with drive-time labels between blocks.

**Emergency reshuffle (`POST /api/schedule/reschedule-emergency`)**:
- Triggered when a `priority = 'emergency'` job needs same-day slotting into an already-full schedule.
- Logic: identify the technician with the lowest-priority job remaining today that could safely move to tomorrow (never move another emergency or a job with a fixed customer-confirmed time window), propose: "Move {low-priority job} to tomorrow and insert {emergency job} at {time}?" вЂ” present this as a single confirm/reject action to the dispatcher, do not silently auto-move other customers' jobs.

### Module 4: Job Management

- Job card (`/jobs/[id]`) aggregates everything tied to a job: linked estimate, appointment time(s), assigned technician, checklist, photos by type (before/progress/after/issue), and the job report.
- Checklist items can be templated per `trade_category` (a simple `job_checklist_templates` lookup the office can edit in settings, copied onto each new job's `job_checklist_items` at creation) so technicians aren't starting from a blank list every time.
- **AI Job Summary** вЂ” the technician records a voice note on `/tech/jobs/[id]` describing what was done; transcribe with Whisper, then format with Claude:
```
System prompt: You are converting a tradesperson's spoken, informal job-completion
note into a professional, concise job report for the customer file. Keep all factual
content (parts used, work performed, any issues found or recommended follow-up).
Remove filler words and informal language. Write in third person, past tense.
Keep it under 100 words unless the original note genuinely requires more detail.
```
- Store both `voice_transcript` (raw) and `ai_formatted_report` (cleaned) on `job_reports` вЂ” always keep the raw transcript accessible via an expander, never discard it.
- Signature capture is a full-screen canvas component (`SignaturePad`) вЂ” export as a PNG, upload to Supabase Storage, save the URL on `job_reports.signature_url`.

### Module 5: AI Estimating Engine

This reuses TradeQuote AI's proven extraction engine and prompt design as its foundation вЂ” do not rewrite it from scratch, port and extend it.

**Core extraction** (voice/text/photo) вЂ” same three-tab pattern as TradeQuote AI's Feature 12 (Speak / Type / Photo), same system prompt structure for structured line-item extraction with server-side recalculation of all money values (never trust LLM arithmetic вЂ” recompute every total, subtotal, VAT, and grand total in code after parsing the AI's response).

**Historical Estimate Engine**: when building the user prompt for extraction, always inject the company's 5 most recent *accepted* estimates in the same `trade_category` as pricing anchors (same pattern as TradeQuote AI's `pricing_history` injection), explicitly instructing the model to stay within roughly 20% of those anchors unless the job description clearly describes a different scope of work.

**Win Probability Score** вЂ” computed after an estimate is generated, before it's sent:
```
System prompt: Given this estimate's price, the company's historical win rate for
similar jobs (same trade_category, similar price band), and how this price compares
to the company's recent average for this job type, estimate the probability (as a
percentage) that the customer accepts this quote. Also return 2-3 short factors
driving that number. Respond with ONLY JSON:
{"win_probability": number (0-100), "factors": ["string", "string"]}
```
- Feed it real aggregated stats computed in code (win rate for this trade_category over the last 90 days, this estimate's price percentile vs. recent accepted estimates) вЂ” do not let the model invent the base rate, calculate it server-side and pass it in.
- Display as a badge on the estimate (`WinProbabilityBadge`: green в‰Ґ70%, amber 40вЂ“69%, red <40%) visible to the contractor only, never to the customer.

### Module 6: Materials Management

- Inventory list (`/materials`) shows each material with a stock-level bar (red when `quantity_on_hand <= reorder_threshold`).
- **AI Inventory Forecasting** (`POST /api/materials/forecast`, run weekly via cron): analyse `materials_used` arrays from recent `job_reports` plus `estimates.line_items` for upcoming scheduled jobs, project likely consumption over the next 30 days per material, flag anything projected to run out before it would naturally be reordered. This is a straightforward consumption-rate calculation done in code (sum recent usage, divide by time period, project forward) вЂ” use Claude only to write the human-readable summary of the forecast, not to do the arithmetic.
- **Supplier "integration"**: be realistic about what's buildable in v1 вЂ” major suppliers (Home Depot, Lowe's, local trade suppliers) do not generally expose public ordering APIs for third-party platforms. Build this as a structured request-drafting workflow instead of live ordering: `material_requests` are created from a low-stock flag or manually, rendered as a clean, professional request document/email ("Materials needed: {list}, requested by {company_name}, needed by {date}") and sent via the supplier's `contact_email` through Resend, or shown as a copyable summary if no email is on file. This delivers the real value (organised, fast reordering) without overpromising live API integration that doesn't exist for most suppliers.

### Module 7: Technician App (PWA)

- Installable via "Add to Home Screen" вЂ” send the install link by SMS/WhatsApp automatically when a team invite is accepted by someone with `role = 'technician'`.
- Built with `next-pwa`: cache the shell and the current day's job list for offline viewing (a technician in a basement with no signal should still see today's jobs and customer address); photo and voice-report uploads queue locally and retry automatically when connectivity returns (use the Background Sync API where supported, with a visible "will upload when back online" indicator as a fallback).
- **AI Copilot** (`/tech/copilot`): a scoped chat вЂ” system prompt restricts it to trade knowledge and troubleshooting only ("You help tradespeople with practical, safe, on-the-job questions about their trade. You do not have access to scheduling, pricing, or customer financial data, and you do not provide it even if asked вЂ” redirect those questions to the office."). Supports photo attachment for visual troubleshooting questions (e.g. "what's wrong with this part") using the same vision-capable model as the photo estimate feature.

### Module 8: Invoicing

- Same core pattern as TradeQuote AI's invoicing feature: one-click "Create invoice" from a completed job (pulls line items from the linked estimate, or lets the office build a fresh line-item list if there wasn't a prior estimate), Stripe Payment Link generation, PDF generation reusing the shared `lib/pdf/generatePdf.ts` template with an "INVOICE" header variant.
- **0.5% platform fee** on payments processed through Stripe Connect вЂ” implement via `application_fee_amount` on the Payment Link/Checkout Session, disclosed in `/settings/billing` as a plain-language line: "FieldOS AI takes a 0.5% fee on payments processed through the platform. Bank transfer and cash payments marked manually are never charged a fee."
- **AI Invoice Builder**: a small voice/text input on the invoice detail page ("Add a line item or adjust this invoice...") вЂ” e.g. "Add an emergency call-out fee and 2 extra metres of pipe." Parse via Claude into a structured line-item diff (`{"add": [...], "modify": [...], "remove": [...]}`), apply it, recalculate totals server-side, and show the contractor a clear before/after diff to confirm before saving вЂ” never silently apply an AI edit to a financial document without a confirm step.
- Overdue handling: extend the existing `follow-up`/cron pattern вЂ” daily check flips `status` to `overdue` past `due_date` and notifies the office via WhatsApp.

### Module 9: Finance Intelligence

- `/finance` profit dashboard: tabs for Job / Technician / Service (trade_category) / Customer profitability, each computed from `job_profitability` (populated by a trigger/queued job whenever an invoice tied to a job is marked paid: `revenue = invoice.total_inc_vat`, `labour_cost` = sum of assigned technician hours Г— their `hourly_rate`, `material_cost` = sum of `job_reports.materials_used` costs, `profit = revenue - labour_cost - material_cost - overhead_allocated`).
- `overhead_allocated` in v1 is a simple even split of a manually entered monthly overhead figure (settings field) divided across the month's completed jobs вЂ” flag this plainly in the UI as an estimate ("overhead is allocated evenly across jobs this month вЂ” for exact figures, consult your accountant"), do not present it as precise accounting.
- **AI CFO** (`/finance/ai-cfo`): build this as Claude with tool-calling against a small, fixed set of safe, read-only, parameterised data-fetching functions вЂ” not free-form SQL generation. Example tools: `get_profit_trend(months)`, `get_technician_profitability(period)`, `get_revenue_by_category(period)`, `get_quote_loss_reasons(period)`. The model decides which tool(s) to call based on the owner's question, the tools run scoped, company-filtered queries server-side, and the model narrates the result in plain language. This keeps the feature both safe (no injection risk, no cross-tenant data leakage) and genuinely useful, rather than hallucinating numbers.
- **Revenue forecasting**: a simple trailing-12-month trend + seasonal adjustment computed in code (linear regression or moving average with month-over-year seasonality factor вЂ” this is a statistics problem, solve it with actual math in TypeScript, not by asking an LLM to "predict" numbers), with Claude used only to write the narrative summary around the chart ("Revenue is trending up 8% month over month, driven mainly by...").

### Module 10: AI Sales System

- **Quote follow-up**: identical mechanism to TradeQuote AI's existing follow-up cron (48hr and 96hr WhatsApp/SMS nudges on unanswered estimates), unchanged.
- **Upselling**: surfaced via the customer detail page upsell panel (Module 1) and, after a job is marked complete, an optional prompt to the office: "Suggest a maintenance plan to {customer_name}?" with a one-tap send of a templated message.
- **Renewal Engine**: `renewal_plans` rows are created either manually or automatically when an estimate/job's `trade_category` and description match a renewal-eligible pattern (e.g. boiler service, gutter cleaning) вЂ” a simple keyword/category rule table in code, editable later. A daily cron checks `next_due_date <= today + 14 days` and sends a reminder message to the customer, then updates `next_due_date` by `interval_months` once the resulting job is completed.

### Module 11: Reputation Management

- **Review requests**: identical mechanism to TradeQuote AI's existing review-request feature вЂ” triggered on invoice paid or manual trigger, WhatsApp/SMS message with the `google_business_profile_url`, configurable delay.
- **Review recovery**: be realistic about data access вЂ” Google does not provide a simple, broadly-available API for third-party apps to monitor a business's incoming review content in real time. Build this as: (a) if/when Google Business Profile API access is approved for the account (a manual application process, not guaranteed), poll for new reviews and auto-trigger recovery outreach on anything в‰¤3 stars; (b) regardless of API access, give the office a manual "Log a bad review" quick-action that immediately triggers the same recovery flow (a personal outreach message draft to the customer, plus an internal flag on their customer profile) вЂ” this makes the feature useful from day one even before/without API approval, rather than being blocked on it.

### Module 12: AI Business Coach

The flagship differentiator вЂ” build this with the same safe tool-calling architecture as the AI CFO (Module 9), but with a broader toolset spanning operations, not just finance: `get_quote_loss_reasons`, `get_technician_performance`, `get_lead_conversion_funnel`, `get_pricing_trend`, `get_seasonal_demand_pattern`, plus everything exposed to the AI CFO. One shared underlying chat engine (`lib/ai/businessCoach.ts`) powers both `/coach` and `/finance/ai-cfo` вЂ” the CFO page is simply the same engine scoped to a financial-only subset of tools and a narrower system prompt, not a separate build.
- `/coach` UI: full-page chat, conversation history in a left rail (one company can have many saved conversations per user), suggested question chips above the input that change based on what's actually notable in the data that week (e.g. surface "Why did your win rate drop in March?" as a suggested chip only if win rate actually dropped that month вЂ” generate these chips from the same `pricing_nudges` logic, don't hardcode generic chips).
- Every coach response that cites a number must be traceable to an actual tool call result, not invented вЂ” log `data_used` (the tool call results) alongside each assistant message in `ai_coach_messages` for debuggability and trust.

### Phase 2 (explicitly NOT in v1): Marketplace

Documented here for completeness, not to be built now. Once FieldOS AI has meaningful contractor adoption in a region, a customer-facing job-matching marketplace ("request a job, get matched to a vetted contractor") becomes viable вЂ” but only because by then there's contractor supply to match demand against. Building it before that point creates a marketplace with no one on one side of it. Do not scaffold marketplace tables, routes, or UI in v1.

## Authentication and roles

- Supabase Auth, email/password + Google OAuth, same as TradeQuote AI.
- Roles (on `team_members.role`): `owner` (full access including billing and account deletion), `admin` (full access except billing/deletion), `dispatcher` (schedule, jobs, leads, calls вЂ” no finance, no team management), `estimator` (estimates, customers, leads вЂ” no schedule/finance/team), `technician` (Technician App only вЂ” `/tech/*` routes; if a technician account tries to access `/app/*` directly, redirect to `/tech/today`), `viewer` (read-only across everything visible to admin, no write actions anywhere).
